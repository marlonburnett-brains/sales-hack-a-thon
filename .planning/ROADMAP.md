### 🚧 v1.8 Structure-Driven Deck Generation (In Progress)

**Milestone Goal:** Close the 5 gaps between the intelligence layer (DeckStructure + element maps) and the generation layer so all touches (1-4) produce visually diverse, design-preserved decks assembled from multiple source presentations using DeckStructure blueprints, context-aware slide selection, and element-map-guided surgical modifications.

**Wave-Based Parallel Execution:**
- Wave 1: Phase 50 (foundation)
- Wave 2: Phase 51 + Phase 52 + Phase 53 (parallel -- three independent tracks)
- Wave 3: Phase 54 + Phase 55 (parallel -- each depends on one Wave 2 track)
- Wave 4: Phase 56 + Phase 57 (parallel -- integration, depend on all prior)

- [ ] **Phase 50: Foundation Types & Interfaces** - Shared TypeScript types and Zod schemas for the generation pipeline
- [ ] **Phase 51: Blueprint Resolver** - DeckStructure consumption producing GenerationBlueprint with candidate slides per section
- [ ] **Phase 52: Multi-Source Slide Assembler** - Cherry-pick slides from multiple source presentations into one deck via copy-and-prune
- [ ] **Phase 53: Modification Planner** - LLM-driven per-slide modification planning using element maps and deal context
- [ ] **Phase 54: Section Matcher** - Context-aware slide selection scoring candidates by industry, pillar, persona, funnel stage
- [ ] **Phase 55: Modification Executor** - Execute planned modifications via scoped Google Slides API batchUpdate
- [ ] **Phase 56: HITL Integration** - Wire 3-stage HITL (Skeleton/Low-fi/High-fi) to the generation pipeline data flow
- [ ] **Phase 57: Touch Routing & Fallback** - Route all 4 touches through structure-driven pipeline with graceful legacy fallback

## Phase Details

### Phase 50: Foundation Types & Interfaces
**Goal**: Every generation pipeline component can import shared types and schemas without circular dependencies
**Depends on**: Nothing (first phase)
**Requirements**: FR-1.1, FR-1.2, FR-1.3, FR-1.4, FR-1.5, FR-1.6, NFR-5
**Success Criteria** (what must be TRUE):
  1. `generation/types.ts` exports GenerationBlueprint, SectionSlot, SlideSelectionPlan, MultiSourcePlan, ModificationPlan, and DealContext types
  2. All LLM-facing schemas are flat objects with no optionals or unions (Gemini structured output compatible)
  3. Types compile cleanly and are importable from any `generation/` module without circular dependency errors
**Plans**: TBD

### Phase 51: Blueprint Resolver
**Goal**: The system can read a DeckStructure and produce an ordered GenerationBlueprint with candidate slides resolved to full SlideEmbedding records
**Depends on**: Phase 50
**Requirements**: FR-2.1, FR-2.2, FR-2.3, FR-2.4, FR-2.5, FR-2.6
**Success Criteria** (what must be TRUE):
  1. Given a touchType + artifactType key, the resolver reads DeckStructure.structureJson and returns a GenerationBlueprint with ordered SectionSlots
  2. Each SectionSlot contains candidateSlideIds resolved to SlideEmbedding records with classification metadata and source presentationId (via Template join)
  3. All 7 logical DeckStructure keys are supported (touch_1, touch_2, touch_3, pre_call, touch_4 x proposal/talk_track/faq)
  4. Missing or empty DeckStructure returns null gracefully (enables downstream fallback routing)
**Plans**: TBD

### Phase 52: Multi-Source Slide Assembler
**Goal**: The system can take a set of slide selections (slideId + source presentationId pairs) and assemble them into a single Google Slides presentation preserving original designs
**Depends on**: Phase 50
**Requirements**: FR-4.1, FR-4.2, FR-4.3, FR-4.4, FR-4.5, FR-4.6, FR-4.7, FR-4.8, FR-4.9, NFR-3, NFR-6
**Success Criteria** (what must be TRUE):
  1. Given slides from 2+ source presentations, the assembler produces a single output presentation with slides from all sources in the specified order
  2. Primary source slides (largest group) are copied via drive.files.copy and pruned -- retaining 100% design fidelity
  3. Single-source case skips multi-source logic and uses standard copy-and-prune efficiently
  4. All temporary Drive copies are cleaned up in finally blocks regardless of success or failure
  5. Output presentation is saved to the deal's Google Drive folder and shared with org
**Plans**: TBD

### Phase 53: Modification Planner
**Goal**: The system can examine a slide's element map and deal context to produce a surgical modification plan specifying which text elements to change and what the new content should be
**Depends on**: Phase 50
**Requirements**: FR-5.1, FR-5.2, FR-5.3, FR-5.4, FR-5.5, FR-5.6, NFR-5
**Success Criteria** (what must be TRUE):
  1. Given a slide's SlideElement records and deal context, the planner produces a ModificationPlan listing specific elements to modify with current and new content
  2. Only text-bearing elements (text, shape with text) are targeted for modification; images, tables, and groups are preserved
  3. The planner distinguishes deal-specific content (company names, industry references) from structural content (methodology, capabilities) and only modifies deal-specific content
  4. A "modification-planner" named agent is registered in AgentConfig with a versioned system prompt
  5. When element maps are missing for a slide, the planner falls back to placeholder injection
**Plans**: TBD

### Phase 54: Section Matcher
**Goal**: For each section in a GenerationBlueprint, the system selects the best candidate slide based on deal context (industry, pillar, persona, funnel stage)
**Depends on**: Phase 51
**Requirements**: FR-3.1, FR-3.2, FR-3.3, FR-3.4, FR-3.5, FR-3.6
**Success Criteria** (what must be TRUE):
  1. Candidate slides are scored using classificationJson metadata (industry, pillar, persona, funnel stage match) against deal context
  2. Vector similarity (pgvector cosine distance) serves as a tiebreaker when metadata scores are equal
  3. Each section's selected slideId resolves to a source presentationId via SlideEmbedding.templateId -> Template.presentationId
  4. When deal context is sparse, the matcher falls back to the highest-confidence candidate rather than failing
  5. Slides already used in prior touches for the same deal are excluded from selection (cross-touch exclusion)
**Plans**: TBD

### Phase 55: Modification Executor
**Goal**: Planned modifications are executed surgically on assembled slides via Google Slides API without cross-slide contamination
**Depends on**: Phase 53
**Requirements**: FR-6.1, FR-6.2, FR-6.3, FR-6.4, NFR-7, NFR-8
**Success Criteria** (what must be TRUE):
  1. Each modification targets a specific element within a specific slide using pageObjectId-scoped batchUpdate operations
  2. The presentation is re-read after each slide's modifications to handle objectId drift
  3. Failed element modifications are skipped with warnings logged, and remaining slides continue processing
  4. No global replaceAllText is used -- all text operations are element-targeted (deleteText + insertText)
**Plans**: TBD

### Phase 56: HITL Integration
**Goal**: Sellers review and approve generation output at three stages: blueprint composition, assembled deck, and modification plan
**Depends on**: Phase 51, Phase 52, Phase 53, Phase 54, Phase 55
**Requirements**: FR-7.1, FR-7.2, FR-7.3, FR-7.4, FR-7.5, FR-7.6, FR-7.7
**Success Criteria** (what must be TRUE):
  1. Skeleton stage presents the GenerationBlueprint with selected slides per section, including thumbnails and match rationale, and the seller can swap selections, toggle optional sections, and reorder sections
  2. Low-fi stage presents the assembled multi-source Google Slides deck URL for review, and the seller can approve or request changes
  3. High-fi stage presents the modification plan summary showing which elements change, executes approved modifications, and presents the final deck URL
  4. All HITL stages use the existing Mastra suspend/resume pattern -- only data payloads change, not the mechanism
**Plans**: TBD

### Phase 57: Touch Routing & Fallback
**Goal**: All 4 touch types route through the structure-driven pipeline when a DeckStructure exists, with graceful fallback to legacy paths when it does not
**Depends on**: Phase 51, Phase 52, Phase 53, Phase 54, Phase 55, Phase 56
**Requirements**: FR-8.1, FR-8.2, FR-8.3, FR-8.4, FR-8.5, FR-8.6, FR-9.1, FR-9.2, FR-9.3, FR-9.4, NFR-1, NFR-2, NFR-4
**Success Criteria** (what must be TRUE):
  1. Touch 1 (pager) routes through blueprint -> single-source assembly -> modifications when DeckStructure exists
  2. Touch 2 (intro deck) and Touch 3 (capability deck) route through blueprint -> multi-source assembly -> modifications when DeckStructure exists
  3. Touch 4 (proposal/talk track/FAQ) routes through blueprint -> multi-source assembly -> modifications when DeckStructure exists
  4. When no DeckStructure exists, each touch falls back to its legacy generation path (slide-assembly.ts, deck-customizer.ts, deck-assembly.ts respectively)
  5. No existing generation code is deleted -- all legacy paths are preserved as fallbacks
  6. DeckStructure confidence gating: green (6+ examples) auto-generates; yellow/red shows warning and offers manual selection
**Plans**: TBD

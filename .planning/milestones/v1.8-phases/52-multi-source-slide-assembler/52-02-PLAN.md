  <action>
    Implement the multi-source path of `assembleMultiSourceDeck`. The function signature accepts:
    ```typescript
    interface AssembleMultiSourceParams {
      plan: MultiSourcePlan;
      targetFolderId: string;
      deckName: string;
      ownerEmail?: string;
    }
    ```

    Implementation steps (following research Pattern 1 + Pattern 2):
    1. Copy primary source via `drive.files.copy({ fileId: plan.primarySource.presentationId, requestBody: { name: deckName, parents: [targetFolderId] } })`
    2. Read primary copy via `slides.presentations.get`, identify slides to delete (not in keepSlideIds)
    3. Batch delete unneeded primary slides via `batchUpdate` with `deleteObject` requests
    4. For each secondary source in `plan.secondarySources`:
       a. Copy secondary source via `drive.files.copy` (track ID in `tempFileIds` array for cleanup)
       b. Read secondary copy via `presentations.get`
       c. For each needed slideId, find the slide in the secondary copy
       d. Create a new slide in the target via `createSlide` request
       e. Read text elements from secondary slide's page elements
       f. Inject text content into the new slide via `insertText` requests scoped to the new slide's objectId
       g. Maintain a mapping `{ originalSlideId -> newSlideId }` for final reorder
    5. Re-read target presentation to get current slide state
    6. Translate `finalSlideOrder` using the objectId mapping (primary IDs preserved, secondary IDs mapped)
    7. Reorder via `updateSlidesPosition`
    8. Call `shareWithOrg({ fileId: targetPresentationId, ownerEmail })`
    9. Return `{ presentationId, driveUrl }`

    CRITICAL: All secondary temp copies tracked in `tempFileIds: string[]` and deleted in a `finally` block (NFR-6). Log warnings on cleanup failures but do not throw.

    CRITICAL: After any batchUpdate that adds/removes slides, re-read via presentations.get before using objectIds (Pitfall 1 from research).

    CRITICAL: All text operations scoped to specific pageObjectIds -- no global replaceAllText (anti-pattern from research).

    Rate limit note (NFR-3): For a typical 12-slide deck from 3 sources, expect ~15-20 API calls total. Batch multiple deleteObject requests into single batchUpdate calls. This is well within the 60 writes/min limit.

    Mock strategy for tests: Use vi.mock for google-auth.ts (getDriveClient, getSlidesClient) and drive-folders.ts (shareWithOrg). Return mock objects that track calls. Verify call sequences and arguments.

<objective>
Validate all 10 LLM schemas against the live Gemini API via round-trip testing: send a realistic domain prompt with each schema, receive structured JSON, and parse it through the Zod schema.

Purpose: Catch any schema rejection errors or Gemini output mismatches in isolation before any agent logic (Phase 4-11) depends on these schemas. This is the phase's core deliverable — schemas that are proven to work with Gemini, not just schemas that compile.

Output: A validation script that runs via `pnpm validate-schemas` in apps/agent, testing all 10 LLM schemas against Gemini 2.5 Flash.
</objective>

  <action>
    Create `apps/agent/src/validation/validate-schemas.ts` following RESEARCH.md Pattern 4 (Gemini API Call with Schema).

    **Structure:**
    1. Import `GoogleGenAI` from `@google/genai`
    2. Import `env` from `../env` for GEMINI_API_KEY
    3. Import `zodToGeminiSchema` and all 10 LLM schemas from `@lumenalta/schemas`
    4. Create `GoogleGenAI` client with env.GEMINI_API_KEY
    5. Define a `SchemaTest` interface: `{ name: string; schema: z.ZodType; prompt: string }`
    6. Define an array of 10 test cases, one per LLM schema

    **Test prompts (realistic domain prompts that mirror actual pipeline usage):**

    1. **TranscriptFieldsLlmSchema:** Extract structured fields from a sample meeting transcript (~100 words covering healthcare patient data interoperability, CTO stakeholder, Q3 timeline, ~$500K budget). Instruction: "Return all fields. Use empty string for any field not found."

    2. **SalesBriefLlmSchema:** "Given the following extracted transcript fields, generate a structured sales brief for Lumenalta. Company: Meridian Health Systems, Industry: Health Care, Subsector: Telehealth. The customer wants a unified patient data platform across 12 clinic locations. Primary capability area: digital transformation. Budget ~$500K, timeline Q3 next year."

    3. **SlideAssemblyLlmSchema:** "You are assembling a proposal slide deck for a healthcare digital transformation engagement. Create a 4-slide deck structure with title, problem statement, proposed solution, and next steps. Each slide should have a title, 3-4 bullet points, speaker notes, and reference a source block ID (use placeholder IDs like 'atlusai-block-001')."

    4. **ROIFramingLlmSchema:** "Generate ROI framing for 2 use cases: (1) Patient data interoperability across 12 clinics, (2) Telehealth platform modernization. For each, provide 2-3 ROI outcome statements and 1 value hypothesis."

    5. **PagerContentLlmSchema:** "Generate content for a first-contact 1-2 pager for Meridian Health Systems in the Health Care industry. Focus on digital transformation capabilities. Include a headline, value proposition, 3-5 key capabilities, and a call to action."

    6. **IntroDeckSelectionLlmSchema:** "You are selecting slides for a Meet Lumenalta intro deck for a Financial Services client. From the following available slide IDs [slide-001, slide-002, slide-003, slide-004, slide-005, slide-006], select the most relevant 4 slides, order them, and provide personalization notes."

    7. **CapabilityDeckSelectionLlmSchema:** "Select slides for a capability alignment deck. Client industry: Technology, Media & Telecommunications. Capability areas: cloud migration, data analytics. Available slide IDs: [cap-001, cap-002, cap-003, cap-004, cap-005, cap-006, cap-007]. Select and order the most relevant slides."

    8. **CompanyResearchLlmSchema:** "Research the company 'Meridian Health Systems' for a pre-call briefing. They are a mid-size healthcare company. Generate key initiatives, recent news items, financial highlights, industry position, and relevant Lumenalta solutions."

    9. **HypothesesLlmSchema:** "Generate role-specific hypotheses for a pre-call briefing. The buyer is a CTO at a healthcare company focused on digital transformation. Create 3 hypotheses, each with evidence and a mapped Lumenalta solution."

    10. **DiscoveryQuestionsLlmSchema:** "Generate 5 prioritized discovery questions for a meeting with a CTO at a healthcare company. Map each question to a specific Lumenalta solution area. Include priority (high/medium/low) and rationale."

    **Execution loop:**
    - For each test case, call `zodToGeminiSchema(test.schema)` to get the JSON Schema
    - Call `ai.models.generateContent()` with model `"gemini-2.5-flash"`, the test prompt, `responseMimeType: "application/json"`, and `responseJsonSchema: jsonSchema`
    - Parse the response text with `JSON.parse()`, then validate with `test.schema.parse()`
    - Log `PASS: {name}` on success, `FAIL: {name} - {error}` on failure
    - Track failures count
    - Add a 500ms delay between Gemini calls to avoid rate limiting

    **Exit behavior:**
    - `process.exit(0)` if all pass
    - `process.exit(1)` if any fail
    - Print summary at end: "X/10 schemas validated successfully"

    **Error handling:**
    - Wrap each test in try/catch
    - On Gemini API error: log the error, increment failure count, continue to next schema
    - On Zod parse error: log the raw Gemini response (first 500 chars) and the Zod error, increment failure count, continue
    - If GEMINI_API_KEY is not set: print clear error message and exit 1 immediately

    **Gemini API pattern (use responseJsonSchema, NOT responseSchema):**
    Per RESEARCH.md, use the new `responseJsonSchema` config property (not the legacy `responseSchema` + `Type` enum). This accepts standard JSON Schema directly from `z.toJSONSchema()` output.

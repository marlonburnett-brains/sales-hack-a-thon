<objective>
Build the shared slide selection and assembly engine that powers Touch 2 and Touch 3 (and later Phase 8). This includes AI-driven content retrieval from AtlusAI, Gemini-based slide selection/ordering, the cross-presentation deck assembly pipeline, salesperson/customer customization injection, and the AtlusAI re-ingestion pipeline for knowledge base growth.

Purpose: This plan creates the reusable infrastructure layer that Touch 2/3 workflows consume. By building it independently of the workflow wiring, the engine can be tested in isolation and the modules remain Touch-agnostic.
Output: Four library modules: slide selection, deck customization, AtlusAI search, and ingestion pipeline.
</objective>

    **AtlusAI search wrapper (apps/agent/src/lib/atlusai-search.ts):**

    1. Create a module that provides slide content search against the knowledge base. Since AtlusAI MCP tools require Claude Code's internal auth and cannot be called from standalone Node.js, use a **Drive API fallback strategy**:
       - searchSlides(params: { query: string, industry?: string, touchType?: string, limit?: number }): Searches the _slide-level-ingestion folder in Drive for Google Docs matching the query. Uses Drive files.list with fullText contains query, filtered by folder. Reads document content via Drive files.export (text/plain). Returns array of { slideId, textContent, speakerNotes, metadata, relevanceScore }.
       - searchByCapability(params: { capabilityAreas: string[], industry: string, limit?: number }): Same approach but constructs a query combining capability areas and industry.
       - Both functions use getDriveClient() with supportsAllDrives: true.

    2. Export a SlideSearchResult interface: { slideId: string, documentTitle: string, textContent: string, speakerNotes: string, metadata: Record<string, unknown>, presentationId?: string, slideObjectId?: string }

    3. Parse the document title to extract presentationId and slideObjectId from the `[SLIDE] PresentationName - Slide N [documentId]` format used by atlusai-client.ts ingestion. Also parse the document description (JSON) for metadata.

    **AI-driven slide selection (apps/agent/src/lib/slide-selection.ts):**

    4. Create selectSlidesForDeck(params: { touchType: 'touch_2' | 'touch_3', companyName: string, industry: string, capabilityAreas?: string[], context?: string, priorTouchOutputs?: string[], limit?: number }) that:
       - Calls searchSlides or searchByCapability depending on touchType
       - Passes search results + deal context to Gemini 2.5 Flash with the appropriate schema:
         - touch_2: IntroDeckSelectionLlmSchema -- Gemini selects and orders slides for an intro deck
         - touch_3: CapabilityDeckSelectionLlmSchema -- Gemini selects slides matching capability areas
       - Uses zodToGeminiSchema for responseJsonSchema
       - Includes priorTouchOutputs in the prompt so AI can reference earlier context (cross-touch continuity per locked decision)
       - Returns the Gemini-selected slide IDs, ordering, and personalization notes

       - Select slides most relevant to the company's industry and context
       - Order slides for a logical narrative flow
       - For Touch 3: prioritize slides matching the specified capability areas
       - Reference prior touch outputs if available to maintain consistency
       - Return only IDs from the provided candidate list (no hallucinated IDs)

    **Deck customizer (apps/agent/src/lib/deck-customizer.ts):**

    1. Create applyDeckCustomizations(params: { presentationId: string, salespersonName?: string, salespersonPhotoUrl?: string, customerName?: string, customerLogoUrl?: string }) that:
       - Builds a batchUpdate request array with replaceAllText entries:
         - `{{salesperson-name}}` -> salespersonName (if provided)
         - `{{customer-name}}` -> customerName (if provided)
       - Adds replaceAllShapesWithImage entries:
         - `{{salesperson-photo}}` -> salespersonPhotoUrl (if provided, CENTER_INSIDE)
         - `{{customer-logo}}` -> customerLogoUrl (if provided, CENTER_INSIDE)
       - Note per Pitfall 1 from research: image URLs must be publicly accessible. If salespersonPhotoUrl is a Drive file ID, construct the public URL or ensure the image has "anyone with link" sharing.
       - Executes batchUpdate on the presentation via getSlidesClient()
       - Returns void (modifies in place)

    2. Create assembleDeckFromSlides(params: { sourcePresentationId: string, selectedSlideIds: string[], slideOrder: string[], targetFolderId: string, deckName: string, customizations?: { salespersonName?, salespersonPhotoUrl?, customerName?, customerLogoUrl? } }) that:
       - Implements the "copy entire source, delete unwanted" strategy per Research recommendation:
         (a) Copy the source presentation to the target folder via Drive files.copy
         (b) Read all slides from the copy via presentations.get
         (c) Identify slides to keep (matching selectedSlideIds by objectId or by content matching)
         (d) Delete slides NOT in the selected list via batchUpdate with deleteObject requests
         (e) Reorder remaining slides to match slideOrder via batchUpdate with updateSlidesPosition
         (f) Apply customizations via applyDeckCustomizations
         (g) Call makePubliclyViewable for iframe preview
       - Returns { presentationId, driveUrl }
       - This is the key function for Touch 2/3 where slides are selected from a source deck

    3. Handle the edge case where selectedSlideIds reference slides from MULTIPLE source presentations (Touch 3 with AtlusAI + L2 capability decks). Strategy: for multi-source assembly, copy the primary template, then for each additional source, extract slide content via presentations.get and recreate using createSlide + insertText requests. Document this limitation clearly in comments -- the single-source "copy and prune" approach handles 90% of cases.

    **AtlusAI re-ingestion pipeline (apps/agent/src/lib/ingestion-pipeline.ts):**

    4. Create ingestGeneratedDeck(params: { presentationId: string, deckName: string, dealContext: { companyName, industry, touchType, decision }, driveFolderId: string }) that:
       - Reads the generated/uploaded presentation via presentations.get
       - Extracts text content from each slide using the existing slide-extractor.ts extractSlidesFromPresentation()
       - For each slide, creates a SlideDocument with:
         - metadata including the deal context (companyName, industry, touchType)
         - metadata.decision = 'approved' | 'overridden' | 'edited' (so AtlusAI can weight positive vs negative signals)
         - metadata.generatedBy = 'system' | 'seller' based on decision
       - Calls ingestDocument() from atlusai-client.ts for each slide
       - Returns { slidesIngested: number, errors: string[] }
       - This enables knowledge base growth: approved decks become positive examples, overrides become learning material for future generation

    5. Add a convenience function shouldIngest(decision: string): boolean that returns true for 'approved', 'edited', and 'overridden' -- all outcomes feed back into the knowledge base per the locked decision "Approved (unmodified) AI pagers are ALSO ingested into AtlusAI as positive examples".

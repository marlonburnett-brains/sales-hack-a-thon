<objective>
Create the data layer (Prisma models) and AI inference engine for deck structures. The system gathers classified examples and templates per touch type, feeds slide descriptions, element maps, classification metadata, and positional data to Google GenAI structured output, and produces a per-touch-type deck structure showing section flow, variations, and mapped reference slides. A cron job auto-re-infers when underlying data changes.

Purpose: This is the backend foundation that powers the entire Deck Intelligence feature. Without inference results stored in the DB, the UI has nothing to display.
Output: DeckStructure + DeckChatMessage Prisma models, inference function, cron job, API endpoints for reading structures.
</objective>

       - `DeckStructure`: id (cuid), touchType (String, @unique), structureJson (String -- JSON of sections array), exampleCount (Int, default 0), confidence (Float, default 0), chatContextJson (String? -- summarized chat constraints for re-inference), dataHash (String? -- hash of example IDs + classifications for change detection), inferredAt (DateTime), lastChatAt (DateTime? -- for active session protection from cron), createdAt, updatedAt.        - `DeckChatMessage`: id (cuid), deckStructureId (String), role (String -- "user" | "assistant"), content (String), structureDiff (String? -- JSON diff if assistant message caused updates), createdAt. Relation to DeckStructure with onDelete Cascade. Index on deckStructureId.

If migration history drift causes failure, create migration SQL manually with CREATE TABLE IF NOT EXISTS and use `prisma migrate resolve --applied` (per CLAUDE.md forward-only discipline).

    3. **Create deck-structure-schema.ts** with:
       - `DeckSection` TypeScript interface: { order: number, name: string, purpose: string, isOptional: boolean, variationCount: number, slideIds: string[] }
       - `DeckStructureOutput` interface: { sections: DeckSection[], sequenceRationale: string }
       - `DECK_STRUCTURE_SCHEMA` constant: Google GenAI `responseSchema` using Type.OBJECT/ARRAY/STRING/NUMBER/BOOLEAN matching the research pattern (see RESEARCH.md Pattern 4). Include descriptive field descriptions so the LLM understands what each field means.
       - `calculateConfidence` function: Takes exampleCount, returns { score: number (0-100), color: "green" | "yellow" | "red", label: string }. Thresholds: 0 examples = 0/red/"No examples", 1-2 = 20+10*n/red/"Low confidence", 3-5 = 50+5*n/yellow/"Medium confidence", 6+ = min(95, 50+7*n)/green/"High confidence".
       - Export all types and constants.

    4. **Create infer-deck-structure.ts** with:
       - `inferDeckStructure(touchType: string, chatConstraints?: string): Promise<DeckStructureOutput>` function that:
         a. Queries all Templates where contentClassification = "example" AND touchTypes JSON array contains the touchType. Also queries templates with contentClassification = "template" for the same touchType (secondary data).
         b. For each example template, loads all SlideEmbeddings with their descriptions and SlideElements (include: { elements: true }).
         c. Builds a detailed prompt including: slide descriptions (purpose, content, use cases), element maps (layout structure per slide), slide position in source deck, classification metadata. Examples are marked as PRIMARY (drive section flow pattern), templates as SECONDARY (expand variation pool). If chatConstraints provided, include as "User Refinement Constraints" section in prompt.
         d. Calls Google GenAI with `responseSchema: DECK_STRUCTURE_SCHEMA` for structured output (use established pattern from classify-metadata.ts -- `ai.models.generateContent` with Vertex AI config).
         e. Parses response, calculates confidence from example count, upserts DeckStructure record via Prisma (upsert on touchType unique constraint). Stores structureJson as JSON.stringify of the output, updates exampleCount, confidence, inferredAt, dataHash.
         f. Returns the parsed DeckStructureOutput.
       - `computeDataHash(touchType: string): Promise<string>` helper that computes a hash (crypto.createHash('sha256')) of sorted example template IDs + their classification data for the given touch type. Used by cron for change detection.
       - Use the same Google GenAI initialization pattern as classify-metadata.ts (import from existing shared AI client or create inline).
       - Prompt engineering: instruct the LLM to analyze the examples as complete decks, identify the common section flow pattern, note which sections are always present vs optional, map specific slide IDs to each section, and explain the sequencing rationale. For templates, note available slide variations that could fill each section.

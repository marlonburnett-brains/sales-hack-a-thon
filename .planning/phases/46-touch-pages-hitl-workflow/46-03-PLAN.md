  <name>Task 2: Wire touch page with full HITL workflow</name>
  <files>
    apps/web/src/app/(authenticated)/deals/[dealId]/touch/[touchNumber]/page.tsx

  <action>
    Replace the placeholder touch page with the full HITL experience. This is a server component that fetches data and renders client components.

    **Server-side data fetching:**
    - Fetch the Deal with interactions filtered by touchType (`touch_${touchNumber}`)
    - Sort interactions by createdAt desc. The most recent interaction is the "active" run.
    - Extract the active interaction's hitlStage, stageContent, status, id, and the workflow runId
    - Pass deal context (companyName, industry) and all interaction data to the client component

    **Create a client component `TouchPageClient` (inline in the file or separate):**
    This is "use client" and manages the interactive state.

    State management:
    - `currentStage`: derived from active interaction's hitlStage (or null if no interaction)
    - `completedStages`: computed from hitlStage (e.g., if current is "lowfi", completed = {"skeleton"})
    - `isGenerating`: tracks whether a new generation is in progress (poll workflow status)
    - `isApproving`: tracks whether stage transition is in flight

    Flows:
    1. **No interactions (not_started):** Render `TouchGuidedStart`. On generate click, call the appropriate `generateTouch*Action` from touch-actions.ts, then poll status using the existing `checkTouch*StatusAction` pattern (reuse polling logic from touch-1-form.tsx). Show `GenerationProgress` during generation.

    2. **Active interaction with hitlStage:** Render `TouchPageShell` with `HitlStageStepper`. Inside the shell, render `TouchStageContent` with the stage content from InteractionRecord.stageContent (parsed from JSON). For Touch 4, `TouchStageContent` delegates to `Touch4ArtifactTabs`.

    3. **Approve stage:** Call `transitionStageAction` with the interaction data. On success, refresh the page data (router.refresh() or revalidate). The workflow resumes server-side, produces next stage content, and the page re-renders showing the new stage.

    4. **Back-navigation (revert):** When user clicks a completed stage in the stepper, call `revertStageAction`. This updates the InteractionRecord and clears downstream stage content. Then trigger a new generation from that stage. Per locked decision: "downstream stages regenerate."

    5. **Generate Another:** When the active interaction has hitlStage="ready" (all stages completed), show a "Generate Another" button. Clicking it starts a new workflow run, creating a new InteractionRecord. The previous one moves to history.

    6. **History:** Pass all interactions except the active one to `TouchGenerationHistory`.

    **Wrap everything in `TouchContextProvider`:**
    Provide the current touch state (touchNumber, touchType, currentStage, stageContent, runId, interactionId) so the Phase 45 chat bar can read it.

    **Touch name mapping:**
    - Touch 1: "First Contact Pager"
    - Touch 2: "Meet Lumenalta"
    - Touch 3: "Capability Alignment"
    - Touch 4: "Sales Proposal"

    **Important:** Keep `export const dynamic = "force-dynamic"` at the top. Keep the VALID_TOUCH_NUMBERS validation.

    **Polling pattern:** Reuse the polling approach from existing touch forms. After starting a workflow, poll every 2 seconds using the appropriate checkTouch*StatusAction. When the workflow status shows a suspend point, stop polling and display the stage content. Use a ref for the polling interval and clean up on unmount.
  </action>

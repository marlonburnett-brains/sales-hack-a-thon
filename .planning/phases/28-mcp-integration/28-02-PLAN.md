  <name>Task 1: Refactor atlusai-search.ts with MCP search routing and LLM extraction</name>
  <files>apps/agent/src/lib/atlusai-search.ts</files>
  <action>
Refactor `atlusai-search.ts` to route through MCP first, falling back to Drive. The file structure becomes:

**1. Update SlideSearchResult interface:**
Add two optional fields:
- `source?: "mcp" | "drive"` -- indicates result origin for UI fallback indicator
- `relevanceScore?: number` -- MCP results get relevance score, Drive results get undefined

**2. Rename existing searchSlides to searchSlidesDrive:**
- Rename the current `searchSlides()` function to `searchSlidesDrive()` (keep it as-is internally)
- Make it NOT exported (internal use only)

**3. Create searchSlidesMcp():**
- Internal function (not exported)
- Import `callMcpTool`, `getCachedExtractionPrompt`, `setCachedExtractionPrompt` from `./mcp-client`
- Build MCP tool args: `{ query: searchText }` where searchText combines query + industry + touchType (same as Drive version)
- If `ATLUS_PROJECT_ID` is set, add `project_id` to args (SRCH-06) -- the MCP tool may or may not accept this; include it and let the server ignore if unsupported
- Call `callMcpTool("knowledge_base_search_semantic", args)` -- note: callMcpTool adds the `atlus_` prefix
- Pass raw result to `extractSlideResults()` (the LLM extraction layer)
- Return results with `source: "mcp"` added to each

**4. Create extractSlideResults() -- LLM extraction layer:**
- Internal async function
- Uses the adaptive prompt pattern from RESEARCH.md Pattern 3
- Import `GoogleGenAI` from `@google/genai` and `env` for project/location
- On first call (no cached prompt): build a discovery prompt that includes:
  - The raw MCP result JSON (stringified, truncated to 8000 chars to stay within LLM context)
  - The target `SlideSearchResult` interface definition (as text)
  - Instructions to extract an array of SlideSearchResult objects from the raw results
  - Instructions to assign a `relevanceScore` (0-1) based on semantic match quality if discernible
  - The search query for context
- On subsequent calls: use the cached prompt template + just the raw results and query
- LLM call: `ai.models.generateContent()` with `model: "openai/gpt-oss-120b-maas"`, `responseMimeType: "application/json"`
- Parse response as `SlideSearchResult[]`
- After first successful extraction: build a cached prompt template from the discovered structure and call `setCachedExtractionPrompt()`
- Batch all results in a single LLM call (user decision: reduce API overhead)
- On LLM failure: log error, return empty array (don't crash the search)

**5. Create new searchSlides() -- the routing function:**
- This is the public exported function with the same signature
- Logic:
  ```
  const useMcp = env.ATLUS_USE_MCP !== 'false';
  if (useMcp && isMcpAvailable()) {
    try {
      return await searchSlidesMcp(params);
    } catch (err) {
      console.warn('[search] MCP search failed, falling back to Drive:', err);
      // Fall through to Drive
    }
  }
  const results = await searchSlidesDrive(params);
  return results.map(r => ({ ...r, source: 'drive' as const }));
  ```
- Import `isMcpAvailable` from `./mcp-client`

**6. searchForProposal() -- NO CHANGES to orchestration (SRCH-03):**
- The multi-pass logic stays identical
- It calls `searchSlides()` internally which now routes through MCP
- Claude's discretion: keep the 3-pass structure as-is. Semantic search may return more relevant results, but the multi-pass strategy provides diversity (pillars, case studies) that a single query can't replicate.

**7. searchByCapability() -- NO CHANGES needed (SRCH-04):**
- Already delegates to `searchSlides()` internally
- Will automatically use MCP through the new routing

**8. Keep Drive imports but make them conditional:**
- The `getDriveClient` import and Drive-specific code stays in `searchSlidesDrive()`
- No code deletion -- Drive is the fallback path

**Key design notes:**
- The `callMcpTool` wrapper handles auth errors, refresh, and rotation internally (from Plan 28-01). This function just calls it and handles the result.
- LLM extraction is always used for MCP results (user decision: consistency over cost savings)
- If both MCP and Drive fail, return empty array with a console.error (Claude's discretion on total failure -- empty results is safest, avoids blocking workflows)

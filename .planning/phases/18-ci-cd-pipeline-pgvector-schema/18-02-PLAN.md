  <action>
Create `.github/workflows/deploy.yml` with the following structure. This is a single workflow triggered on push to main with 4 jobs in sequence:

**Job 1: `checks`** (runs-on: ubuntu-latest)
- Checkout with fetch-depth: 2
- Setup pnpm@9 via pnpm/action-setup@v3
- Setup Node 22 via actions/setup-node@v4 with cache: 'pnpm'
- `pnpm install --frozen-lockfile`
- `pnpm turbo lint build` (this runs lint + type-check + build for all packages per turbo config)

**Job 2: `migrate`** (needs: checks, runs-on: ubuntu-latest)
- Checkout
- Setup pnpm@9 + Node 22 + cache (same as checks)
- `pnpm install --frozen-lockfile`
- `npx prisma migrate deploy` with working-directory: apps/agent
- env: DATABASE_URL from secrets.DATABASE_URL
- If this job fails, deploy-agent and deploy-web CANNOT run (enforced by `needs:`)

**Job 3: `deploy-agent`** (needs: migrate, runs-on: ubuntu-latest)
- Use container: ghcr.io/railwayapp/cli:latest
- Checkout
- `railway up --service ${{ vars.RAILWAY_SERVICE_ID }} --ci`
- env: RAILWAY_TOKEN from secrets.RAILWAY_TOKEN

NOTE on Railway container: The `ghcr.io/railwayapp/cli:latest` container may not have `git` available. Since `actions/checkout@v4` needs git, use the alternative approach if the container doesn't work:
- runs-on: ubuntu-latest (not container)
- Install Railway CLI: `npm install -g @railway/cli`
- Then run `railway up --service ${{ vars.RAILWAY_SERVICE_ID }} --ci`

Choose whichever approach works. The container approach is preferred per research but fall back to npm install if needed.

**Job 4: `deploy-web`** (needs: deploy-agent, runs-on: ubuntu-latest)
- Checkout
- Setup pnpm@9 + Node 22 + cache
- `pnpm install --frozen-lockfile`
- `npm install --global vercel@latest`
- `vercel pull --yes --environment=production --token=${{ secrets.VERCEL_TOKEN }}`
- `vercel build --prod --token=${{ secrets.VERCEL_TOKEN }}`
- `vercel deploy --prebuilt --prod --token=${{ secrets.VERCEL_TOKEN }}`
- env: VERCEL_ORG_ID from secrets.VERCEL_ORG_ID, VERCEL_PROJECT_ID from secrets.VERCEL_PROJECT_ID

IMPORTANT design decisions (per user's locked decisions):
- The user specified "Sequential deploy order: migrate -> agent -> web". deploy-web uses `needs: [deploy-agent]` to enforce the full chain: checks -> migrate -> deploy-agent -> deploy-web.

Workflow trigger: `on: push: branches: [main]`

Add `concurrency` group to prevent parallel workflow runs on rapid pushes:
```yaml
concurrency:
  group: deploy-${{ github.ref }}
  cancel-in-progress: false
```
(cancel-in-progress: false ensures a running deploy finishes rather than being cancelled)

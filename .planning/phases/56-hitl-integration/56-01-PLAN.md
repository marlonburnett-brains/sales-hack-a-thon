<objective>
Wire the 3-stage HITL workflow (Skeleton / Low-fi / High-fi) to the structure-driven generation pipeline built in Phases 50-55.

Purpose: This is purely an orchestration layer. All hard logic (blueprint resolution, slide selection, multi-source assembly, modification planning, modification execution) already exists in Phase 51-55 modules. The workflow steps are thin wrappers that call these functions, shape data for suspend payloads, and thread seller-approved changes downstream.

Output: A single `structure-driven-workflow.ts` with 7 steps (resolve+select, skeleton suspend, assemble, lowfi suspend, plan mods, highfi suspend, execute+record) registered in Mastra, with unit tests covering all suspend/resume contracts.

Create `apps/agent/src/generation/structure-driven-workflow.ts` following the exact Mastra workflow pattern from touch-1-workflow.ts:

**Workflow input:** `{ dealId: string, touchType: string, artifactType?: string }`
**Workflow output:** `{ interactionId: string, presentationId: string, driveUrl: string, decision: string }`

**7 Steps:**

1. `resolveAndSelectSlides` (computation step, no suspend):
   - Query Deal + Company from prisma for DealContext assembly
   - Call `resolveBlueprint({ touchType, artifactType }, dealContext)` -- if null, throw (Phase 57 handles routing; this workflow assumes DeckStructure exists)
   - Call `selectSlidesForBlueprint(blueprintWithCandidates)`
   - Create InteractionRecord with `touchType`, `status: "in_progress"`, `hitlStage: "skeleton"`
   - Store slim stageContent (sections + selections with thumbnails, NOT full candidate objects)
   - Return: blueprint, selections, candidates map (for thumbnail lookup), interactionId, dealContext fields

2. `awaitSkeletonApproval` (suspend step):
   - suspendSchema: `{ stage: "skeleton", interactionId, dealId, blueprint: { touchType, sequenceRationale, sections: [{ sectionName, purpose, isOptional, selectedSlideId, sourcePresentationId, candidateSlideIds }] }, selections: [{ sectionName, slideId, matchRationale, thumbnailUrl }] }`
   - resumeSchema: `{ decision: "approved" | "refined", refinedBlueprint?: { sections: [{ sectionName, isOptional, selectedSlideId, candidateSlideIds }] } }`
   - On "refined": update SectionSlot selectedSlideIds from refinedBlueprint, re-derive SlideSelectionPlan from modified sections, update stageContent
   - Pass approved selections downstream

3. `assembleMultiSourceDeckStep` (computation step):
   - Get deal folder (getOrCreateDealFolder pattern from touch-1)
   - Build MultiSourcePlan via `groupSlidesBySource` + `buildMultiSourcePlan` using approved selections
   - Call `assembleMultiSourceDeck({ plan, targetFolderId, deckName, ownerEmail })`
   - Update hitlStage to "lowfi", stageContent = `{ presentationId, driveUrl }`
   - Return: presentationId, driveUrl, slideCount, interactionId, dealContext

4. `awaitLowfiApproval` (suspend step):
   - suspendSchema: `{ stage: "lowfi", interactionId, dealId, presentationId, driveUrl, slideCount }`
   - resumeSchema: `{ decision: "approved" | "request_changes" }`
   - On "request_changes": Return a signal that the workflow should restart from skeleton. Per research recommendation, this loops back by returning `{ restart: true }`. The caller (Phase 57 routing) handles re-invocation. For now, throw an error with a descriptive message indicating restart is needed, since Mastra workflows don't natively support loopback. Keep it simple.
   - On "approved": pass through

5. `planAndPrepareModifications` (computation step):
   - For each slide in the approved selection, get the slideObjectId from the assembled presentation (re-read presentation to get current slide objectIds)
   - Call `planSlideModifications({ slideId, slideObjectId, dealContext })` for each slide
   - Collect ModificationPlan[] results
   - Build modificationSummary (per-slide: slideId, modificationCount, elements with elementId/reason/preview)
   - Update hitlStage to "highfi", stageContent = slim modificationSummary (NOT full plans)
   - Return: presentationId, driveUrl, modificationPlans[], modificationSummary, interactionId

6. `awaitHighfiApproval` (suspend step):
   - suspendSchema: `{ stage: "highfi", interactionId, dealId, presentationId, driveUrl, modificationSummary }`
   - resumeSchema: `{ decision: "approved" | "rejected" }`
   - On "rejected": update hitlStage to "ready" with no modifications applied, return final URL as-is
   - On "approved": pass modificationPlans downstream

7. `executeAndRecordFinal` (computation step):
   - Call `executeModifications({ presentationId, plans: modificationPlans })`
   - Update InteractionRecord: status=approved, decision=approved, hitlStage="ready", driveFileId=presentationId, outputRefs=[driveUrl]
   - Create FeedbackSignal with source "structure_driven_approve"
   - Return: interactionId, presentationId, driveUrl, decision

**Chain:** `.then(resolveAndSelectSlides).then(awaitSkeletonApproval).then(assembleMultiSourceDeckStep).then(awaitLowfiApproval).then(planAndPrepareModifications).then(awaitHighfiApproval).then(executeAndRecordFinal).commit()`

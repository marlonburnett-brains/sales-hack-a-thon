  <action>
    1. Add to apps/web/src/lib/api-client.ts (at the bottom, following existing pattern):
       ```typescript
       // ────────────────────────────────────────────────────────────
       // Google Token Storage (Phase 22 -- OAuth Scope Expansion)
       // ────────────────────────────────────────────────────────────

       export async function storeGoogleToken(data: {
         userId: string;
         email: string;
         refreshToken: string;
       }): Promise<{ success: boolean; tokenId: string }> {
         return fetchJSON<{ success: boolean; tokenId: string }>("/tokens", {
           method: "POST",
           body: JSON.stringify(data),
         });
       }

       export async function checkGoogleToken(
         userId: string
       ): Promise<{ hasToken: boolean }> {
         return fetchJSON<{ hasToken: boolean }>(`/tokens/check/${userId}`);
       }
       ```

    2. Modify apps/web/src/app/auth/callback/route.ts:
       - Import `storeGoogleToken` from "@/lib/api-client"
       - CRITICAL: Capture the FULL return from `exchangeCodeForSession` — change from `const { error }` to `const { data, error }` because `data.session.provider_refresh_token` is only available from this return value, NOT from subsequent getSession() calls (see RESEARCH.md Pitfall 2).
       - After domain enforcement passes and before the redirect, check for `data.session.provider_refresh_token`:
         ```typescript
         const refreshToken = data.session?.provider_refresh_token;
         if (refreshToken && user?.id && user?.email) {
           try {
             await storeGoogleToken({
               userId: user.id,
               email: user.email,
               refreshToken,
             });
           } catch (err) {
             console.error("[auth] Failed to store Google token:", err);
             // Set search param to trigger toast after redirect
             const redirectUrl = new URL(`${origin}${next}`);
             redirectUrl.searchParams.set("token_error", "1");
             return NextResponse.redirect(redirectUrl.toString());
           }
         }
         ```
       - If no refreshToken is present (user didn't consent or returning user), allow login normally — the middleware will flag them for re-consent on next visit.
       - Keep existing domain enforcement logic exactly as-is.

    3. Note on `data.session` vs `getUser()`: The callback currently calls `getUser()` after `exchangeCodeForSession` for domain enforcement. This is fine — keep the `getUser()` call for the user object, but extract `provider_refresh_token` from the `data.session` returned by `exchangeCodeForSession` (not from getUser or getSession).

<task type="auto">
  <name>Task 3: Middleware re-consent detection with cookie cache</name>
  <files>apps/web/src/middleware.ts</files>
  <action>
    1. After the `getUser()` call that already exists in middleware, and ONLY for authenticated users NOT on /login or /auth pages, add a token existence check.
    2. Use a cookie cache approach (per RESEARCH.md recommendation, CONTEXT.md Claude's discretion):
       - Check for cookie `google-token-status` on the request
       - If cookie value is "valid": skip the check, proceed normally
       - If cookie value is "missing": sign out user, redirect to /login?reconsent=1
       - If cookie is absent: make a fetch call to agent `GET /tokens/check/:userId`
         - Use `process.env.AGENT_SERVICE_URL` and `process.env.AGENT_API_KEY` for the fetch (middleware runs in Edge, cannot import server-only api-client.ts, so make a direct fetch)
         - If hasToken is true: set cookie `google-token-status=valid` with maxAge=3600 (1 hour) on the response
         - If hasToken is false: set cookie `google-token-status=missing` with maxAge=3600, then sign out and redirect to /login?reconsent=1&next={returnTo}
         - If fetch fails (agent down): log warning and proceed without check (graceful degradation -- don't block users if agent is unreachable)
    3. IMPORTANT: The cookie must be set on the supabaseResponse object (the response that flows through middleware), not on a new response.
    4. Skip the token check entirely for paths starting with /login, /auth, or static assets (already excluded by matcher but double-check).
    5. When the auth callback successfully stores a token, it should also set the `google-token-status=valid` cookie so the middleware doesn't immediately re-check. Add this to the callback redirect response in the auth callback file as well (coordinate with Task 1 output).
  </action>

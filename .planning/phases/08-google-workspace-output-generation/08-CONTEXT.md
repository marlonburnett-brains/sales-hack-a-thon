# Phase 8: Google Workspace Output Generation - Context

**Gathered:** 2026-03-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Given a validated SlideJSON (Phase 7 output), create a fully formatted Google Slides deck, a slide-by-slide talk track Google Doc, and a buyer FAQ Google Doc in a named per-deal folder in shared Lumenalta Drive — reusing the slide assembly infrastructure from Phase 4. Building the HITL-2 review UI (Phase 9), new web app pages, or new API endpoints is out of scope — this phase delivers the three Workspace artifacts as workflow steps only.

</domain>

<decisions>
## Implementation Decisions

### Slides deck assembly
- Template copy + populate pattern (reuses assembleFromTemplate() from Phase 4)
- Retrieved slides (sourceType="retrieved"): copy the original Google Slides slide from its source presentation using sourceBlockRef to locate it, preserving original visual layout and formatting. Inject bespoke copy via Slides API batchUpdate text replacements
- Synthesized slides (sourceType="synthesized"): use branded section templates — maintain a branded template deck with one template slide per section type (title_context, problem_restatement, roi_outcomes, next_steps). Copy the matching template slide into the deck, then inject synthesized content via text replacement
- All artifacts go in the existing per-deal folder (getOrCreateDealFolder() from Phase 4) — no subfolders
- Deck naming: "[CompanyName] - [PrimaryPillar] - [Date]" per success criteria
- Deck made publicly viewable (makePubliclyViewable()) for iframe preview, consistent with Phase 4

### Talk track Google Doc
- Slide-by-slide speaker notes format — one section per slide
- Reuse the speakerNotes already generated by Phase 7's generateCustomCopy step — no additional Gemini call needed
- Simple headings + body text format: H1 = deck title, H2 = each slide title, body = speaker notes text under each H2
- Doc naming: "[CompanyName] - [PrimaryPillar] Talk Track - [Date]" — consistent with deck naming pattern
- Created via Google Docs API (getDocsClient() already exists)

### Buyer FAQ Google Doc
- Role-specific objections: generate objections per stakeholder role from the approved brief (e.g., CIO concerns about integration risk, CFO concerns about ROI timeline)
- 2-3 objections per stakeholder role, giving 6-12 total for a typical brief with 3-4 stakeholders
- Single Gemini call with approved brief context (stakeholders, business outcomes, constraints, pillars) to generate all objections + responses
- New BuyerFaqLlmSchema Zod schema for structured output
- Doc structure: H1 = doc title, H2 = each stakeholder role, bold = objection text, body paragraph = recommended response
- Doc naming: "[CompanyName] - [PrimaryPillar] Buyer FAQ - [Date]"

### Workflow integration
- 3 new sequential steps appended to touch-4-workflow (currently 11 steps):
  - Step 12: createSlidesDeck — core deliverable, depends on SlideJSON from step 11
  - Step 13: createTalkTrack — depends on deck slide order from step 12
  - Step 14: createBuyerFAQ — logically last, uses brief data
- Drive URLs (deckUrl, talkTrackUrl, faqUrl) stored in InteractionRecord.outputRefs after all 3 artifacts are created
- Workflow output schema updated to: deckUrl, talkTrackUrl, faqUrl, slideCount, dealFolderId (SlideJSON no longer in output — consumed by step 12)
- No new API endpoints — Phase 8 is purely workflow steps. Phase 9 will add review UI and artifact display endpoints

### Claude's Discretion
- How to handle slides where source presentation is unavailable (fallback to section template)
- Exact batchUpdate request sequencing for slide copying and text injection
- Google Docs API request structure for heading/paragraph styling
- BuyerFaqLlmSchema field definitions
- Error handling for Google API failures during artifact creation
- How to read live objectIds from copied slides for text replacement targeting

</decisions>

<specifics>
## Specific Ideas

- Retrieved slides preserve original visual fidelity — the deck looks like it was assembled from real Lumenalta decks, not generated from scratch
- Talk track is a quick-reference prep doc sellers scan before the meeting — not a coaching manual
- FAQ grouped by stakeholder role so sellers can flip to the section matching whoever they're meeting
- Sequential artifact creation (deck -> talk track -> FAQ) keeps the workflow simple and debuggable
- Drive URLs in outputRefs means the deal page timeline can show direct links to all three artifacts immediately

</specifics>

- Gemini 2.5 Flash structured output: zodToGeminiSchema + JSON parse + Zod .parse() round-trip

- Slides API: batchUpdate with replaceAllText for text injection, presentations.get for objectId discovery

- JSON string serialization for complex objects in workflow step passthrough (Mastra nested schema storage limitation)

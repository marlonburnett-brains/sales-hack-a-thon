  <action>
**`/api/drive/token` route** (GET):
- Authenticate the current user from the Supabase session (use `createClient()` from `@/lib/supabase/server`)
- Call the agent's existing token-cache system to get a fresh Google access token for this user. Two approaches in priority order:
  1. Call the agent API endpoint that returns a fresh access token (check if one exists)
  2. If no such endpoint exists, create a simple proxy that calls the agent's `/api/drive/token` endpoint (the agent has access to UserGoogleToken refresh tokens)
- If approach 2 is needed, also create a minimal agent-side route that uses `getPooledGoogleAuth(userId)` to get a fresh access token and returns it as JSON `{ accessToken: string }`
- Return `{ accessToken: string }` on success, 401 if not authenticated, 500 on error
- This solves research pitfall #6: provider_token is not persistent after login

**settings-actions.ts** (server actions):
- `getDriveRootFolder(userId: string): Promise<{ folderId: string | null; folderName: string | null }>` -- Query UserSetting for keys "drive_root_folder_id" and "drive_root_folder_name" by userId. Return both or null if not set.
- `setDriveRootFolder(userId: string, folderId: string, folderName: string): Promise<void>` -- Upsert both UserSetting records (drive_root_folder_id and drive_root_folder_name) for the user. Use prisma's upsert with the @@unique([userId, key]) constraint.
- These server actions call the agent API to read/write UserSetting since Prisma client is in the agent app. Use the existing `agentFetch()` or direct fetch pattern to agent API.

**env.ts** (web app):
- Add `NEXT_PUBLIC_GOOGLE_API_KEY: z.string().min(1)` for the Picker API browser key
- Add `NEXT_PUBLIC_GOOGLE_CLOUD_PROJECT: z.string().min(1)` for the Picker appId (or reuse NEXT_PUBLIC_GOOGLE_CLIENT_ID if the project already exposes it)
